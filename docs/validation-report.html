Findings & Improvement Ideas

✅ Improvements shipped since last validation

  1. PeeringDB 429 retry/backoff. Fixed silent IX-overlap failure (PR #2). 429s retry with exponential backoff + jitter, honoring Retry-After. The ## IX Overlap section always renders when --my-asn is supplied — success or "unavailable" — and never silently disappears.
  2. RIPEstat retry/backoff parity. Same retry pattern applied to RIPEstatClient._request. Previously the RIPEstatRateLimitError raise sat inside the try block where the except clauses only caught httpx exceptions — so 429s never retried.
  3. Persistent file-backed cache (no DB). New FileCache stores per-key JSON files under ~/.cache/route-sherlock/ (honors XDG_CACHE_HOME). Zero dependencies, human-readable, clearable with rm -rf. Atomic writes (write-to-temp + rename) survive crashes mid-write.
  4. --offline flag on peer-risk and safeguards. Reuses on-disk cache; on cache miss raises OfflineCacheMiss which the CLI converts into a clear "run once without --offline to populate" message. Time-windowed RIPEstat queries (bgp-updates, routing-history) snap their timestamps to the hour so cache keys are stable across runs — without this, offline re-runs returned wrong scores because the churn-detection data was always a cache miss.
    Verified: online run 14.0s → offline re-run 0.49s, identical MODERATE 72/100 result for AS267613 (T15 PASS).

🟡 Soft expectations / known transient behavior

  1. "Policy: Open" for AS13335 (slide 6) depends on PeeringDB. The tool's primary outputs (risk tier, network name, score) come from RIPEstat and are robust. The peering-policy field requires a successful PeeringDB lookup — under sustained load (e.g., the test suite itself), PeeringDB rate-limits aggressively and the field degrades to "Unknown". The tool's fallback is correct; this is a data-source quirk to note rather than a defect.
  2. Stability fetch can transiently fail while still reporting a maximum stability score. Better to either short-circuit with a clearly labelled "stability: unknown" tile or to retry with the same back-off used for PeeringDB.

🛡 Reliability improvements still worth doing

  1. Persistent on-disk cache (DuckDB or SQLite). The design-v2 doc already names this. Even a 24-hour TTL cache of RIPEstat as-overview, announced-prefixes and PeeringDB net/netixlan would eliminate the vast majority of repeated API calls.
  2. Apply the same retry/backoff to RIPEstat collector. The new logic is currently PeeringDB-only; RIPEstat occasionally returns transient errors that propagate as "Could not fetch stability data: Request failed".
  3. Offline / cached-only flag (--offline). Lets users run scoring from a pre-warmed cache when wifi is flaky.

🚀 Feature additions worth considering after NANOG

  1. RPKI ROA validation in scoring. The deck advertises RPKI-invalid rejection (slide 14) and an RPKI column (slide 16), but the security category does not actually check RPKI. Hook RIPEstat rpki-validation (or routinator HTTP API) into the security score so the deck is fully truthful.
  2. Watch / scheduled re-scoring. Slide 9 explicitly recommends “implement monitoring” for MODERATE peers. A small route-sherlock watch AS<n> --interval 7d --alert-on score-drop>10 with a sqlite store would make that recommendation actionable.
  3. Markdown / PDF report export. Peering coordinators forward decisions over email; --report markdown on peer-risk / safeguards is low-effort and high-value.
  4. Vendor-specific config snippet output on safeguards (--vendor cisco|junos|bird). Originally proposed and dropped — worth revisiting if the workshop audience asks for copy-pasteable BGP config.
  5. Score-component drill-down command. peer-risk --explain that dumps every input metric and the exact reason for each point deduction. Today the deductions are summarised but not auditable.
  6. Bulk mode. route-sherlock peer-risk --batch asns.txt --json for peering coordinators evaluating dozens of candidates.
